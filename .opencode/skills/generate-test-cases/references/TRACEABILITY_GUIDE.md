# Traceability Guide

## 目标

把“设计 / 代码 / 风险 / 测试”串起来，避免只写一堆分散的 case。

## 推荐做法

### 设计优先
从设计文档或 README 中提取：

- 系统目标
- 核心工作流
- 用户路径
- 输入输出约束
- 权限模型
- 错误处理策略

### 代码优先
从代码中提取：

- 对外入口
- public API / CLI / route
- 配置键
- 关键状态对象
- 外部依赖边界
- 异常处理和 fallback

### 风险优先
高优先级常见来源：

- 状态迁移复杂
- 并发或重试
- 鉴权鉴权绕过
- 配置错误
- 外部系统不可用
- 数据损坏或格式异常
- 幂等性与重复提交

## 判断哪些场景值得写成 case

优先保留：

- 直接影响主流程成败
- 容易回归
- 一旦失败代价高
- 之前团队经常漏测
- 与权限、安全或资金相关
